# Product

## Register

product

## Users
Developers and technical teams using Locality to preserve coding-session context, manage local memory, and control which devices can access their private workspace. They use the product while moving between coding agents, repositories, billing, and local AI workflows, usually in the middle of active implementation work.

## Product Purpose
Locality keeps the useful output of coding sessions available after the session ends. It helps users capture prompts, commands, patches, and decisions locally, reconnect devices safely, and query project memory through a local AI bridge without turning private history into a cloud-only dependency. Success means the next session starts with context instead of reconstruction.

## Brand Personality
Private, precise, dependable. The product should feel calm, technically credible, and trustworthy rather than flashy. It should read like an engineering tool that respects user control and keeps sensitive project history local.

## Anti-references
Overdecorated AI dashboards, novelty-heavy chat toys, generic glassmorphism SaaS panels, loud purple gradient "agent" interfaces, and marketing-style surfaces that make routine product tasks feel theatrical. Avoid anything that suggests the tool is guessing, hiding state, or prioritizing visual spectacle over operational clarity.

## Design Principles
1. Keep the task in front: every screen should help the user understand status, next action, and system state without hunting.
2. Preserve trust through restraint: use familiar product patterns, clear hierarchy, and explicit state over decorative effects.
3. Local-first should be visible: privacy, device control, and on-machine memory should feel concrete in the interface, not implied.
4. Continuity matters more than novelty: adjacent dashboard screens should share the same component language, spacing rhythm, and interaction model.
5. Dense information is acceptable when it stays ordered: the user is technical, so compact detail is fine if the layout remains legible and calm.

## Accessibility & Inclusion
Target WCAG AA contrast across light and dark themes. Maintain visible keyboard focus states, readable touch targets, and reduced-motion-safe transitions. Ensure status is not conveyed by color alone, and keep core flows understandable for users moving quickly between keyboard, pointer, and different screen sizes.
